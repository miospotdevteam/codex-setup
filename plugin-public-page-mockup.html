Why it exists

Built against the failure modes that make coding agents expensive.

The plugin is aimed at the familiar problems: shallow exploration, dropped scope, shared-code edits without consumer checks, weak typing, and work that cannot survive compaction. Instead of adding more generic prompt text, it imposes an operating system.

Failure mode Scope drift

Rules that make silent scope cuts explicit and unacceptable.

Failure mode Blind edits

Mandatory exploration, consumer checks, and blast-radius review before touching shared code.

Failure mode Context loss

Plans are written to disk and updated throughout execution so work survives compaction.

Failure mode Weak verification

Completion means typecheck, lint, tests, and the relevant discipline checks are actually run.

Conductor layer

Route the task to the right discipline.

The conductor is the top-level contract. It decides when the work needs planning, frontend design, TDD, systematic debugging, refactoring, or skill creation, then forces the right sequence.

Plan integrity

Make compaction survivable by default.

Persistent plans are not optional. They live in the repo, become the execution source of truth, and are updated often enough that a fresh context window can resume cleanly.

Review gate

Present the plan before implementation begins.

Non-trivial work is meant to pass through Orbit review so the intent is frozen before execution. That makes the implementation legible to both the user and the model.

Skill pack

Codex-native disciplines, not a generic prompt bundle.

The page below treats the skills as modules in a system. Each one solves a recurring failure pattern: planning, debugging, refactoring, TDD, frontend design, persistent recovery, or skill authoring.

Core operating layer

Conductor + engineering discipline

These are the default rails for coding work. They enforce exploration, blast-radius checks, no type shortcuts, and explicit verification.

  • `lbyl-conductor` orchestrates the workflow.
  • `lbyl-engineering-discipline` governs how edits are made.
  • `lbyl-persistent-plans` keeps progress recoverable on disk.
Behavior change layer

Debugging, TDD, refactoring

When the task becomes more specific, the system routes to a narrower discipline rather than relying on vague best-effort behavior.

  • `lbyl-systematic-debugging` requires root-cause work before fixes.
  • `lbyl-test-driven-development` enforces red/green rhythm.
  • `lbyl-refactoring` protects cross-file structural changes.
Interface layer

Frontend and skill authoring

The pack also includes creative and meta-disciplines for building polished interfaces and authoring new reusable skills.

  • `lbyl-frontend-design` prevents generic UI output.
  • `immersive-frontend` handles expressive motion-heavy surfaces.
  • `lbyl-skill-creator` improves or ships new skills.
Workflow

A session rhythm that makes the agent legible.

The plugin turns the session into a sequence that can be inspected: explore first, write the plan to disk, review the plan, execute with the right disciplines, then verify the final result.

1. Discover the real shape of the task Read files, imports, consumers, adjacent patterns, and conventions before implementation begins.
2. Freeze intent in `plan.json` and `masterPlan.md` Persist what will be built, how it will be verified, and where the risk lives.
3. Execute with routed skills Call the right specialty: frontend design, debugging, TDD, refactoring, or skill creation.
4. Finish only after verification Typecheck, lint, tests, and domain-specific review are part of completion, not an optional afterthought.
